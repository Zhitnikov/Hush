# Messenger 'Hush'

Веб-мессенджер: личные чаты, группы и каналы, файлы, голосовые и видеосообщения, звонки через браузер, опросы, реакции, отложенные сообщения. Сервер на Node.js, клиент на React. Сообщения в личке шифруются на клиенте перед отправкой; сервер хранит уже зашифрованный текст (см. раздел про безопасность).

Репозиторий рассчитан на локальный запуск и доработку под себя. Продакшен-настройки (HTTPS, домены, секреты, лимиты) нужно добавлять отдельно.

## Кратко про стек

| Слой | Технологии |
|------|------------|
| Клиент | React 18, Vite 4, Tailwind CSS, Zustand (состояние), Socket.IO client, axios, @tanstack/react-virtual (длинные списки чатов), date-fns, lucide-react (иконки), clsx |
| Сервер | Node.js, Express 4, Socket.IO 4, Mongoose 7 + MongoDB |
| Аутентификация | JWT (jsonwebtoken), пароли через bcryptjs |
| Файлы | Multer, раздача из папки `server/uploads` |
| Криптография в браузере | Web Crypto API (RSA-OAEP, AES-GCM), ключи в localStorage |

Почему так:

- **React + Vite** — быстрая сборка и dev-сервер, без лишней обвязки.
- **Zustand** — простое глобальное состояние без шаблонного кода; разделено на сторы: приложение, сайдбар, окно чата.
- **Socket.IO** — одно соединение для real-time: сообщения, печать, статусы, звонки-сигналы; REST остаётся для списков, истории, загрузки файлов.
- **MongoDB + Mongoose** — документная модель хорошо ложится на сообщения и пользователей; схемы в `server/models`.
- **JWT** — stateless-сессии: клиент хранит токен и шлёт в заголовке, сервер проверяет подпись.
- **Virtualizer** — при большом числе чатов в списке рендерятся только видимые строки, чтобы не тормозить UI.

## Требования

## Переменные окружения (сервер)

В папке `server` можно создать файл `.env`:

| Переменная | Назначение |
|------------|------------|
| `MONGO_URI` | Строка подключения к MongoDB. По умолчанию в коде: `mongodb://localhost:27017/securemessenger` |
| `PORT` | Порт HTTP (по умолчанию `5000`) |
| `JWT_SECRET` | Секрет для подписи JWT. В коде есть запасной вариант для разработки — в бою обязательно свой длинный секрет |

Без `.env` сервер всё равно стартует с дефолтами из `server.js`.

## Запуск

**1. MongoDB** должна быть доступна по URI из `MONGO_URI` или локально на порту 27017.

**2. Сервер**

```bash
cd server
npm install
npm run dev
```

Или `npm start` без автоперезапуска.

**3. Клиент** (отдельный терминал)

```bash
cd client
npm install
npm run dev
```

Vite обычно поднимает фронт на `http://localhost:5173`. API в `client/src/config.js` указывает на `http://localhost:5000` — порты должны совпадать с тем, где реально слушает сервер.

Сборка клиента для статики:

```bash
cd client
npm run build
```

Папка `client/dist` — результат. Раздачу с продакшен-сервера настраиваете сами (nginx и т.д.).

## Как устроена работа

- **Регистрация / вход** — `POST /api/auth/register`, `POST /api/auth/login`. Ответ: JWT и данные пользователя. Клиент кладёт токен в `localStorage` и при необходимости генерирует пару RSA-ключей, публичный ключ отправляет на сервер (`/api/auth/update-key`).

- **Список пользователей и каналов** — REST в `routes/chat.js` (`/users`, `/channels`, папки, поиск и т.д.), с JWT в заголовке `Authorization: Bearer ...`.

- **Сообщения** — основной поток через Socket.IO: `send_message`, `receive_message`, редактирование, удаление, реакции, опросы, печать, прочитанность. Отложенные сообщения: в БД поле `scheduledAt`, фоновый интервал в `server.js` раз в несколько секунд проверяет очередь и рассылает готовые.

- **Каналы** — тип в модели канала (например, группа vs broadcast): в broadcast постить может только создатель, это проверяется в обработчике `send_message`.

- **Файлы** — `POST /api/chat/upload` с multipart, ответ с `fileUrl` и `fileType`; дальше файл уходит в сообщении через сокет как ссылка на `/uploads/...`.

- **Звонки** — WebRTC в `CallInterface`: сигналы (offer/answer/ice) гоняются через Socket.IO (`call_user`, `answer_call`, `webrtc_signal`, `end_call`). Отдельный TURN/STUN в проект не входит — для работы через жёсткий NAT может понадобиться свой сервер ретрансляции.

## Безопасность и шифрование

- В **личных** чатах текст сообщения на клиенте шифруется (Web Crypto), на сервер уходит уже ciphertext. Расшифровать может сторона с приватным ключом. Публичные ключи пользователей хранятся на сервере для обмена.

- **Группы/каналы** — схема с общим групповым ключом (AES), ключи для участников оборачиваются RSA; детали в `client/src/utils/crypto.js` и использовании в `ChatWindow`.

- Сервер **всё равно** видит метаданные: кто кому пишет, размеры, время, вложения по URL. Это не анонимная сеть уровня Tor — обычный сервер-приложение.

- CORS в коде выставлен широко (`origin: '*'`) — для продакшена лучше сузить под свой фронтенд.

- Пароли в БД только в виде хеша (bcrypt).

Итог: это учебный/пет-проектный уровень архитектуры «шифрование на клиенте + хранение на сервере», а не сертифицированный мессенджер. Для реальных угроз нужен отдельный аудит и инфраструктура.
